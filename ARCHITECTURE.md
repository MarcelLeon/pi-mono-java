# Pi-Mono Java 项目架构文档

## 🎯 项目目标

**核心目标**: 为Spring项目和Spring AI大模型应用提供完整的AI代理框架接入方案

**次要目标**:
- 与SpringAI集成，提供统一的LLM抽象层
- 支持多LLM提供商（Anthropic, OpenAI, Google, Azure等）
- 提供企业级稳定性、安全性和监控功能
- 持续同步TypeScript版本的功能

## 🏗️ 项目架构

### 模块结构

```
pi-mono-java/
├── pi-core/                    # 核心接口和模型 - ✅ 完成
│   ├── AgentSession.java       # 会话管理接口
│   ├── SessionNode.java        # 会话节点（ULID + 树结构）
│   ├── LLMProvider.java        # LLM提供者抽象
│   ├── ToolDefinition.java     # 工具定义接口
│   └── ToolCall.java           # 工具调用模型
│
├── pi-session/                 # 会话管理实现 - ✅ 完成
│   ├── SessionTree.java        # 会话树实现
│   ├── SessionManager.java     # 会话管理器
│   └── SessionPersistence.java # JSONL持久化
│
├── pi-llm/                     # LLM集成层 - ✅ 完成
│   ├── LLMProviderManager.java # 提供者管理器
│   ├── mock/MockLLMProvider.java # Mock实现
│   └── [未来: AnthropicProvider.java, OpenAIProvider.java]
│
├── pi-tools/                   # 工具系统 - ✅ 完成
│   ├── ToolManager.java        # 工具管理器
│   ├── ToolPermissionManager.java # 权限管理
│   ├── 内置工具:
│   │   ├── ReadFileTool.java   # 文件读取
│   │   ├── WriteFileTool.java  # 文件写入
│   │   ├── EditFileTool.java   # 文件编辑
│   │   ├── BashTool.java       # Bash命令
│   │   ├── FindFileTool.java   # 文件搜索
│   │   ├── GrepTool.java       # 内容搜索
│   │   └── LsTool.java         # 列出文件
│
├── pi-cli/                     # 命令行界面 - ✅ 完成
│   └── PiCliApplication.java   # CLI入口
│
├── pi-ui/                      # UI组件（待开发）
├── pi-starter/                 # Spring Boot Starter（待开发）
├── pi-test/                    # 测试框架（待开发）
└── pi-extensions/              # 扩展系统（待开发）
```

### 技术栈

- **Java 21** - 最新LTS版本，支持虚拟线程
- **Spring Boot 3.4.0** - 企业级应用框架
- **Spring AI** - 统一的AI集成（独立抽象层）
- **Maven** - 构建和依赖管理
- **ULID** - 时间有序的唯一ID
- **JSONL** - 会话持久化格式
- **Picocli** - 命令行界面
- **Lanterna** - 终端UI（未来）

### 架构设计原则

1. **模块化设计** - 每个模块职责单一，可独立使用
2. **接口抽象** - 核心功能通过接口抽象，便于扩展
3. **依赖注入** - 使用Spring DI，便于测试和集成
4. **安全优先** - 工具执行带沙箱和权限控制
5. **性能优化** - 虚拟线程、内存优化、快速启动
6. **企业级** - 健康检查、监控、日志、错误处理

## 🔄 开发机制

### 反思机制

**触发条件**:
1. 每完成一个主要功能模块后自动触发
2. 每次compact之前强制触发
3. 开发迭代超过3天时提醒触发

**反思内容**:
1. **目标对齐检查** - 当前开发是否服务于核心目标
2. **架构一致性** - 新增功能是否符合架构设计原则
3. **集成便利性** - 是否便于Spring项目接入
4. **测试覆盖度** - 端到端测试是否充分
5. **性能影响** - 是否影响整体性能目标

**记录位置**: `CLAUDE.md` 文件的"开发反思"部分

### 开发流程

```
需求分析 → 架构设计 → 编码实现 → 单元测试 → 集成测试 → 端到端测试 → 反思总结
```

### 端到端测试策略

**测试层次**:
1. **单元测试** - 单个组件功能
2. **集成测试** - 模块间协作
3. **端到端测试** - 完整业务流程

**测试场景**:
1. **Spring Boot集成测试** - 验证starter可用性
2. **LLM切换测试** - 验证多提供商支持
3. **工具链测试** - 验证工具组合使用
4. **会话持久化测试** - 验证数据完整性
5. **性能测试** - 验证响应时间和资源使用
